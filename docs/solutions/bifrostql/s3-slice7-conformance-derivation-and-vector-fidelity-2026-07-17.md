# S3 slice 7: conformance-kit derivation, vector fidelity, dedup discipline — clean run, epic closeout

**Lesson.** S3 epic's final slice (conformance + security matrix + AWS
CLI/rclone fixtures) passed review in one round (0 blockers, 1 advisory).
Three durable patterns, reusable beyond S3:

1. **When a shared conformance kit doesn't fit the adapter's wire shape,
   self-prove the same invariant instead of forcing derivation.**
   `ProtocolAdapterConformanceTests` models a request as table + column-list +
   GraphQL filter over an `IProtocolAdapter`. S3's wire object is a
   bucket/key addressing one file column — no verb projects arbitrary
   columns or inserts a row. Forcing S3 into that shape would fake a request
   the protocol cannot express and prove nothing real. Instead
   `S3AdapterConformanceTests` drives a matrix over every op class
   (list-buckets, list-objects, get, head, put, delete, copy) with a
   tampered signature over an exploding lister/seam: each returns a clean
   403 at the SigV4 gate (which projects identity through the shared
   `IBifrostAuthContextFactory`) and the exploding path never runs — a 403
   instead of the 404/500 the explosion would raise is the structural proof
   the shared seam is unskippable. `ChatEndpointConformanceTests` made the
   same non-derivation call earlier for the same reason (HTTP-carried,
   fixed-shape surface can't express arbitrary-column requests). This is now
   a repeatable judgment call for future adapters: check whether the kit's
   request model can express the adapter's actual wire shape before forcing
   derivation; if not, self-prove the same invariant and document the
   non-derivation decision in the test class, mapping authorized-path proof
   to existing per-op tests.

2. **Interop vectors must be vendor-published, not self-signed.** The
   pre-existing SigV4 tests signed with `S3TestSigner`, which shares the
   verifier's own canonicalization — proving signer and verifier agree with
   each other, never that either agrees with AWS. `S3SigV4PublishedVectorTests`
   fixed this by feeding the verifier AWS's own published S3 examples (GET
   and PUT Object, credential `AKIAIOSFODNN7EXAMPLE`, signing time
   `20130524T000000Z`) with AWS's documented expected signatures, clock
   pinned to the signing instant, plus a tamper-negative proving acceptance
   is genuine signature agreement, not the verifier ignoring the check. Any
   test claiming spec/vendor interop must source its vectors from the
   vendor's published examples — a vector generated by the same code under
   test proves self-consistency, not interop, and is vacuous for the claim
   it's meant to support.

3. **A test-only "already covered elsewhere" claim needs a review technique,
   not a review pass.** The AWS CLI/rclone fixture slice (criterion 4)
   required proving real client request shapes round-trip; the reviewer
   verified the ponytail-dedup claim ("authorized-path seam traversal
   already proven") by checking the class doc's mapping against actual
   existing test file:line references, not by trusting the comment. The
   sole advisory this slice raised — the exploding-double matrix has no
   in-file positive control proving a valid signature reaches the exploding
   seam — was accepted as non-vacuous only because the reviewer independently
   confirmed the discriminating assertion (`Code=="SignatureDoesNotMatch"`)
   AND traced authorized traversal to real middleware tests over the actual
   pipeline. Coverage-dedup claims in test-only slices should be checked
   this way: locate the specific existing test asserting the claimed
   coverage, don't take the comment's word for it.
